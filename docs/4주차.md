## 🛠 오늘 진행한 활동

### 1. 실제 동작 가능한 에이전트 구현 실험 결과 공유

* 각자 지난주까지 진행한 실험을 바탕으로
  **“에이전트가 실제로 행동할 수 있는가?”**에 초점을 맞춰 결과 공유
* 단순한 계획·문서 생성이 아닌,

  * 외부 도구 접근
  * 명령 실행
  * 상태 변화 유발
    이 가능한 구조인지 여부를 중점적으로 논의함

---

### 2. AI Agent vs Agentic AI 개념 정리 및 레퍼런스 공유 (동인)

* **AI Agent**와 **Agentic AI**의 개념 차이를 명확히 구분함

  * AI Agent:

    * 정해진 역할과 범위 내에서
      지시 → 처리 → 결과 반환을 수행
  * Agentic AI:

    * 목표 기반으로 스스로 계획을 세우고
      도구를 선택·조합하여 행동을 시도
* 관련 레퍼런스 영상 공유 및 토론 진행

  * [https://www.youtube.com/watch?v=Ye-ewCPn8EE](https://www.youtube.com/watch?v=Ye-ewCPn8EE)
* 이를 통해 현재 우리가 만들고 있는 구조는
  **완전한 Agentic AI라기보다는,
  “Agent + Tool Orchestration 구조”에 가깝다**는 데 합의함

---

### 3. 실행 한계와 구조적 보완 시도

#### (1) MCP 기반 직접 실행의 한계

* MCP 환경에서 에이전트가

  * 브라우저 조작
  * 외부 서비스 로그인
  * 실제 버튼 클릭 / 입력
    등을 **직접 수행하는 데 한계**가 있음을 재확인
* 단일 에이전트에게 모든 실행 책임을 부여하는 방식은
  현실적으로 불안정함을 인지함

#### (2) Tool Agent 분리 구조 제안

* 이를 보완하기 위해,

  * **행동을 실제로 수행하는 Tool Agent**
  * **의사결정·지시를 담당하는 상위 Agent**
    를 분리하는 구조를 실험함
* 구조 개념:

  * CEO(유저)가 보유한 Tool을
  * 에이전트에게 “직접 위임”하는 것이 아니라
  * **에이전트가 Tool Agent를 통해 실행 요청**
* 특히,

  * HR Agent가 신규 에이전트 생성 시
  * 역할에 맞는 Tool을 할당하는 방식으로
    구조화하는 방향을 테스트함

---

### 4. 실행 실험 사례 및 기술적 한계

* **Playwright**를 활용한 자동화 실험 진행

  * 일부 시나리오에서는 동작 확인
  * 그러나 쿠팡 등 주요 플랫폼에서
    **봇 탐지로 인해 차단**되는 문제 발생
* 이는 단순 구현 문제가 아니라,

  * 상용 서비스 자동화 시
  * 반드시 고려해야 할 **현실적 제약 조건**임을 확인함

---

### 5. 산출물 포맷 개선 시도 (지원)

* 기존 JSON 형태의 결과물을
  **Markdown 기반 산출물**로 개선
* 가독성과 활용성은 향상되었으나,

  * 여전히 “문서 생성” 단계에 머무르고 있으며
  * **실제 실행으로의 직접 연결은 어려운 상태**
* 결과적으로,

  * Output 포맷 개선은 의미 있으나
  * 실행 레이어 문제를 근본적으로 해결하지는 못함을 인지함

## 📌 오늘의 정리

* 에이전트를 실제로 “동작하게” 만드는 것은
  단순히 프롬프트나 사고 구조의 문제가 아님을 명확히 인식함
* 핵심 인사이트는 다음과 같음

  * ✔️ 계획·판단·지시: AI가 충분히 잘함
  * ❌ 외부 세계에서의 직접 행동:
    보안, 인증, 자동화 한계로 인해 구조적 제약 존재
* 따라서,
  **단일 만능 에이전트 모델보다는**

  * 역할별 Agent
  * 실행 전용 Tool Agent
  * 유저(CEO)의 최종 승인·위임
    이 결합된 **현실적인 Hybrid 구조**가 필요함을 확인함
