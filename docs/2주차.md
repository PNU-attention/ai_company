## 🛠 오늘 진행한 활동

### 1. ai_company 사용자 스토리 분석

* CEO, HR, RM(Resource Manager), Expert Agent로 구성된 **AI Company 구조**를 사용자 스토리 관점에서 분석함
* 단순 기능 나열이 아닌,
  **“누가 → 언제 → 어떤 맥락에서 → 무엇을 하는가”** 중심으로 스토리를 재정의함
* 회사 목표 설정부터 최종 결과 보고까지의 **엔드투엔드 흐름**을 정리함

---

### 2. 사용자 시나리오(User Scenario) 정리

* 총 9개의 핵심 사용자 시나리오를 정의함

  * 회사 초기 설정 (CEO Onboarding)
  * 전문가 에이전트 생성 및 확인
  * 목표 기반 프로젝트 분해
  * 프로젝트 실행 계획 수립 및 오케스트레이션
  * 에이전트별 작업 수행
  * 전문성 초과 요청(Escalation)
  * 추가 채용(에이전트 생성)
  * 작업 완료 보고
  * 최종 결과 보고
* 각 시나리오마다 **주체 / 목적 / 흐름 / 결과**를 명확히 구분하여 정리함
* SDD 관점에서 **시나리오가 곧 개발 단위의 출발점**이 됨을 확인함

---

### 3. 기능 명세서(Function Specification) 작성

* 사용자 시나리오를 기반으로 **기능 단위 명세(F-01 ~ F-10)**를 도출함
* 각 기능에 대해 다음 요소를 명확히 정의함

  * 기능 ID
  * 주체(Agent)
  * 입력값(Input)
  * 출력값(Output)
  * 기능 설명 및 역할
* 특히 RM 에이전트를 중심으로 한
  **프로젝트 분해, 작업 오케스트레이션, 순차/병렬 실행 관리** 기능을 핵심으로 설계함

---

### 4. 멀티 에이전트 역할 구조 정립

* ai_company 내 에이전트의 역할을 구조적으로 정리함

  * **CEO**: 목표 정의 및 최종 의사결정
  * **HR Agent**: 목표 기반 전문가 에이전트 생성(에이전트 팩토리)
  * **RM Agent**: 프로젝트 분해, 작업 분배, 실행 오케스트레이션, 리스크 관리
  * **Expert Agent**: 자율적 작업 수행 및 에스컬레이션
* 각 에이전트가 **자율성을 가지되, 상호 의존적으로 협업하는 구조**임을 명확히 함

---

## 📌 오늘의 정리

* ai_company는 단순한 멀티 에이전트 시스템이 아니라,
  **조직 구조와 업무 흐름을 그대로 모사한 AI Company 모델**임을 확인함
* SDD 방법론을 적용함으로써

  * “기능 먼저”가 아닌 **“스토리 → 시나리오 → 기능”** 흐름이 자연스럽게 이어짐
* RM 에이전트의 오케스트레이션 역할이 프로젝트 성공의 핵심임을 인식함
* 초기 단계에서 사용자 시나리오와 기능 명세를 명확히 정리하는 것이
  이후 Claude Code 기반 구현 속도와 품질에 결정적인 영향을 줄 것으로 판단됨
