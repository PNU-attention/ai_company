# ai_company

> **AI 에이전트로 구성된 회사**
> Spec-Driven, Agent-Oriented AI Organization Prototype

---

## 1. 프로젝트 개요

**ai_company** 프로젝트는
AI를 단순한 도구가 아닌 **조직의 구성원(직원)**으로 정의하고,
AI 에이전트들이 역할을 나누어 협업하며 하나의 회사를 운영하는 모델을 실험하는 프로젝트입니다.

본 프로젝트의 핵심 가설:

> CEO(사용자)가 회사의 목표를 정의하면,
> **HR 에이전트가 필요한 전문가를 채용**하고,
> **RM 에이전트가 프로젝트를 분해하여 작업을 조율**하며,
> **Expert 에이전트들이 실제 업무를 수행**하여 목표를 달성할 수 있다.

## 2. 핵심 에이전트 구조

```

              │                           │

      ┌───────────────┐           ┌───────────────┐
      │  HR 에이전트    │           │  RM 에이전트     │
      │ (에이전트 팩토리) │◄───────── │ (오케스트레이션)  │
      └───────────────┘           └───────────────┘
                                          │
              ┌───────────────────────────┼───────────────────────────┐
              │                           │                           │
              ▼                           ▼                           ▼
      ┌───────────────┐           ┌───────────────┐           ┌───────────────┐
      │ Expert Agent 1│           │ Expert Agent 2│           │ Expert Agent N│
      │  (동적 생성)    │           │  (동적 생성)    │           │  (동적 생성)     │
      └───────────────┘           └───────────────┘           └───────────────┘
```

### 역할 분담

| 에이전트 | 역할 | 책임 |
|----------|------|------|
| **CEO** | 의사결정자 | 회사 목표 정의, 최종 승인 |
| **HR** | 에이전트 팩토리 | 전문가 에이전트 생성, 추가 채용 |
| **RM** | 핵심 두뇌 | 프로젝트 분해, 작업 분배, 오케스트레이션, 최종 보고 |
| **Expert** | 실무자 | 할당된 작업 수행, 완료 보고, Escalation |

## 3. 워크플로우

### 3.1 기본 워크플로우 (9단계)

```
CEO → HR/RM → Expert Agents → 📄 문서 산출물 → 👤 사람이 제작 → CEO 컨펌
```

| 단계 | 시나리오 | 주체 | 설명 |
|------|----------|------|------|
| 1 | CEO Onboarding | CEO | 회사 설명/목표 입력 |
| 2 | 전문가 에이전트 생성 | HR | 목표 분석 → 필요 역할 추론 → 에이전트 생성 |
| 3 | 프로젝트 분해 | RM | 목표를 Project 단위로 분해 |
| 4 | 실행 계획 수립 | RM | Task 세분화, 에이전트 배정, Execution Graph 생성 |
| 5 | 작업 수행 | Expert | 할당된 Task 실행, 산출물 생성 |
| 6 | Escalation | Expert → RM | 전문성 초과 시 RM에게 요청 |
| 7 | 추가 채용 | RM → HR | 필요시 신규 에이전트 생성 요청 |
| 8 | 완료 보고 | Expert → RM | Task 완료 결과 보고 |
| 9 | 최종 보고 | RM → CEO | 프로젝트 종합 보고서 전달 |

### 3.2 확장 워크플로우 (MCP 통합)

```
CEO → HR/RM → Expert Agents → 🎨 실제 에셋 생성 → CEO 컨펌 → 🚀 자동 배포
```

기본 워크플로우가 문서 산출물에서 끝나는 반면, 확장 워크플로우는 **MCP(Model Context Protocol)**를 활용하여 실제 에셋을 생성하고 자동 배포까지 수행합니다.

| 단계 | 시나리오 | 주체 | 설명 |
|------|----------|------|------|
| 1-8 | 기본 워크플로우 | - | 기본 워크플로우와 동일 |
| 9 | 에셋 생성 | Expert + MCP | MCP를 통해 실제 디자인/코드/미디어 에셋 생성 |
| 10 | CEO 컨펌 | CEO | 생성된 에셋 검토 및 승인 |
| 11 | 자동 배포 | System + MCP | 승인된 에셋 자동 배포 |

#### 현재 연결된 MCP

| MCP | 용도 | 상태 |
|-----|------|------|
| Figma MCP | UI/UX 디자인 에셋 생성 및 조회 | ✅ 연결됨 |

#### MCP 동적 확장 지침

회사의 목표 달성이나 CEO의 명령 수행을 위해 필요한 MCP가 있다면, **그때그때 동적으로 MCP를 추가**할 수 있습니다:

1. **필요성 판단**: RM 또는 Expert 에이전트가 작업 수행 중 외부 도구 연동이 필요하다고 판단
2. **MCP 요청**: CEO에게 필요한 MCP 연결 요청
3. **MCP 연결**: CEO가 해당 MCP 연결 승인 및 설정
4. **작업 재개**: 연결된 MCP를 활용하여 작업 계속 진행

**예시 MCP 확장 시나리오**:
- 코드 배포가 필요한 경우 → GitHub/GitLab MCP 연결
- 문서 자동 생성이 필요한 경우 → Google Docs/Notion MCP 연결
- 이미지 생성이 필요한 경우 → DALL-E/Midjourney MCP 연결
- 데이터 분석이 필요한 경우 → Database MCP 연결

## 4. 핵심 개념

| 개념 | 설명 | 저장 위치 |
|------|------|-----------|
| **Company Context** | 회사 설명, 목표, 전략 | `company/context.json` |
| **Agent Pool** | 현재 활성화된 에이전트 목록 | `company/agents.json` |
| **Project** | 목표 달성을 위한 오케스트레이션 단위 | `projects/{id}/project.json` |
| **Task** | 최소 실행 단위 | `projects/{id}/tasks/{task-id}.json` |
| **Execution Graph** | Task 간 순서/병렬 관계 | `projects/{id}/execution-graph.json` |

## 6. 개발 철학

### Spec-Driven Development (SDD)

코드는 다음이 준비되기 전까지 작성되지 않습니다:

- 요구사항 명세
- 유저 스토리
- 기술 스펙
- 테스트 케이스

### 파일 기반 결과물

- 모든 산출물은 파일로 저장 (JSON, Markdown)
- Database 불필요
- Git으로 버전 관리

## 7. 차별점

- AI를 **조직 단위 문제 해결 주체**로 정의

- SDD 기반 예측 가능한 개발 흐름
